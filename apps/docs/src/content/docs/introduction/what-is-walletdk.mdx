## What it is

WalletDK is a TypeScript SDK for embedding a **self-custodial Bitcoin, Lightning, and Ark wallet** directly in your web or mobile app. Your users hold their own keys. You do not run a wallet server on their behalf.

The full wallet daemon ([darepo-client](https://github.com/lightninglabs/darepo-client)) is compiled to run **inside your app**: to WebAssembly in the browser on web, and into the app binary on React Native. There is no separate process, no local socket, and no backend you must operate to host the wallet logic. Your app boots the daemon and talks to it through a typed client.

WalletDK ships as these packages:

| Package | Role |
|---|---|
| `@lightninglabs/wavelength-core` | Shared types, the `WalletDKClient` interface, errors, and enums. No DOM, no transport. |
| `@lightninglabs/wavelength-web` | The browser WASM transport. Framework-agnostic: use it from vanilla JS, Vue, Svelte, or React. Re-exports `core`. |
| `@lightninglabs/wavelength-react-native` | The React Native transport. The daemon is compiled into the app binary. Re-exports `core`. |
| `@lightninglabs/wavelength-react` | React provider and hooks. Depends on `core` only and takes an injected client, so the same binding can run over the web or React Native transport. |

**React is the primary developer experience**, but React is not required. On web, if you are not using React, install `@lightninglabs/wavelength-web`, call `createWebClient()`, and use the `WalletDKClient` methods directly. The React package is a thin binding over the same interface.

## Mental model

Think of WalletDK as **your app talking to a local wallet daemon**, not your app talking to a remote wallet API.

1. **Your app** calls methods like `balance()`, `send()`, and `deposit()` on a `WalletDKClient`.
2. **The client** forwards those calls to the embedded daemon running in your app (on web, in a Web Worker by default, or the main thread if configured).
3. **The daemon** holds keys, tracks VTXOs, coordinates Ark rounds, and reaches out to public infrastructure when it needs chain data, mailbox relay, or Lightning swaps.

The daemon connects to **three backend REST gateways** (configured in `RuntimeConfig`):

- **Ark operator + mailbox** (`arkServerUrl`) for rounds and VTXO/mailbox relay
- **Swap server** (`swapServerUrl`) for Lightning↔Ark atomic swaps
- **Esplora** (`esploraUrl`) for chain and UTXO queries

Your app never calls those gateways directly. It only talks to the local daemon. **User keys never leave the device.** Seed generation, signing, and wallet state live on the device (OPFS in the browser; the app data directory on React Native). The gateways see protocol traffic, not your users' private keys.

A minimal boot sequence looks like this:

<PlatformTabs>
<div data-label="Web">

```ts
import { createWebClient, defaultConfig } from '@lightninglabs/wavelength-web';

const client = createWebClient({ runtimeBaseUrl: '/walletdk/' });

await client.ready();                        // load WASM runtime
await client.start(defaultConfig('signet')); // boot daemon, connect gateways
const { mnemonic } = await client.createWallet({ password: '…' }); // show mnemonic to the user for backup
```

</div>
<div data-label="React Native">

```ts
import { createNativeClient, defaultConfig } from '@lightninglabs/wavelength-react-native';

const client = createNativeClient();

await client.ready();                        // ready the native runtime
await client.start(defaultConfig('signet')); // boot daemon, connect gateways
const { mnemonic } = await client.createWallet({ password: '…' }); // show mnemonic to the user for backup
```

</div>

From here, use the same client for deposits, Lightning receive, sends, and activity. The [System architecture](/introduction/system-architecture/) page explains how those operations map to Ark, swaps, and on-chain exits.
