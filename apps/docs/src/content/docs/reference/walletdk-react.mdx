`@lightninglabs/walletdk-react` is the React binding for WalletDK: a context provider that hands a [`WalletEngine`](/reference/walletdk-core/#WalletEngine) down to a component subtree, plus hooks for reading its state and calling wallet operations.

<Callout type="note">
This package re-exports the entire `walletdk-core` surface (`WalletDKClient`, `WalletEngine`, `RuntimeConfig`, `WalletInfo`, `Balance`, `Entry`, `RuntimePhase`, request/result types, and every enum), so you can import core types directly from `@lightninglabs/walletdk-react` instead of adding `@lightninglabs/walletdk-core` as a separate dependency. The engine itself is not built here: create one with `createWebWalletEngine` from [walletdk-web](/reference/walletdk-web/) or `createNativeWalletEngine` from [walletdk-react-native](/reference/walletdk-react-native/). Every hook on this page, including `useWalletPasskey`, must be called within a `WalletDKProvider` subtree; each throws if it is not.
</Callout>

## Mutations

The eight hooks below drive a wallet operation and carry hook-local, verb-prefixed `*Pending`/`*Error`/`*Data` state, all built on one shared convention:

<Callout type="note" title="Mutation convention: throw-and-capture">
Every mutation hook exposes the same shape, with its fields prefixed by the verb it performs: an action function (e.g. `create`), `<verb>Pending: boolean`, `<verb>Error: Error | null`, `<verb>Data` (the last successful result, or `null`), and `reset<Verb>(): void`. Calling the action clears the previous error/data, sets pending to `true`, awaits the underlying engine call, and on completion **both** updates the hook's state **and** settles the returned promise the normal way: it resolves with the result (and sets the data field) or rejects with the failure (and sets the error field to that same `Error` instance). This is throw-and-capture: read the `*Error`/`*Data` fields for a declarative render, or `await`/`try`/`catch` the action's return value for imperative flow, whichever fits the call site. Error fields are always `Error | null`, never a string. `reset<Verb>()` clears the pending/error/data trio back to idle without calling anything.

The verb prefix is deliberate: a component composing several of these hooks (e.g. `create` and `deposit` side by side) never needs destructure aliases to avoid name collisions. [`useWalletPasskey`](#useWalletPasskey)'s `create`/`open` fields follow the same prefixing pattern, but this is one place the prefix does not save you: `useWalletCreate` and `useWalletPasskey` both expose a `create` verb, so destructuring both in the same component collides on all four of `create`, `createPending`, `createError`, and `resetCreate`. Keep one as a namespaced object (e.g. `const passkey = useWalletPasskey(...)`) instead of destructuring both. Each hook's return type is also exported as `UseWallet<Verb>Result` (`UseWalletCreateResult`, `UseWalletRestoreResult`, `UseWalletUnlockResult`, `UseWalletDepositResult`, `UseWalletReceiveResult`, `UseWalletPrepareSendResult`, `UseWalletSendResult`, `UseWalletRefreshResult`), for typing wrapper components around a hook without re-deriving its shape.

A `PasskeyCancelledError` (the user dismissed the OS prompt) is the one exception: it rethrows without being recorded into the error field, since a dismissed prompt is not a failure to display.
</Callout>

## Re-exported from walletdk-core

`@lightninglabs/walletdk-react` re-exports the full `@lightninglabs/walletdk-core` surface (`export * from '@lightninglabs/walletdk-core'`), so `WalletDKClient`, `WalletEngine`, `createWalletEngine`, `RuntimeConfig`, `defaultConfig`, `WalletInfo`, `WalletStatus`, `Balance`, `Entry` and the rest of the activity/exit/send type family, `RuntimePhase`, `WalletState`, `RecoveryState`, `PasskeyCancelledError`, `toError`, and every request/result/enum type are all importable directly from this package. The enumerated values for `RuntimePhase` and `WalletState` are documented on the [walletdk-core reference](/reference/walletdk-core/) rather than repeated here.
