`@lightninglabs/walletdk-react` is the React binding for WalletDK: a context provider that owns a `WalletDKClient` and wires its state into React, plus hooks for reading that state and calling wallet operations.

<Callout type="note">
This package re-exports the entire `walletdk-core` surface (`WalletDKClient`, `RuntimeConfig`, `WalletInfo`, `Balance`, `Entry`, `RuntimePhase`, request/result types, and every enum), so you can import core types directly from `@lightninglabs/walletdk-react` instead of adding `@lightninglabs/walletdk-core` as a separate dependency. The transport is not re-exported: use `createWebClient` from [walletdk-web](/reference/walletdk-web/) or `createNativeClient` from [walletdk-react-native](/reference/walletdk-react-native/). Every hook on this page, including `usePasskeyWallet`, must be called within a `WalletDKProvider` subtree; each throws if it is not.
</Callout>

Provide exactly one of `client` or `createClient`. If both are given, `client` wins. If neither is given, the provider throws synchronously (not just on first render effect) with an error asking for one of the two props.

<Callout type="note">
The provider drives several behaviors automatically so hosts do not each reimplement them:

- **Sync auto-poll**: while `phase` is `'syncing'`, the provider calls `refresh()` every 2000ms until the phase leaves `'syncing'`. After 5 consecutive `refresh()` failures it stops polling, sets the top-level `error`, and moves `phase` to `'error'` instead of polling forever.
- **Activity-driven auto-refresh**: while `phase` is `'ready'`, the provider subscribes to `activity` events (via `startActivity({ includeExisting: true })`) and, on each event, calls `refresh()` after a 250ms debounce. This is how balance and activity stay current without a host-managed poll.
- **Unsolicited `stopped` transition**: a `runtimeStopped` event, whether from a clean `stop()` or a worker crash/fatal surfaced by the transport, moves `phase` to `'stopped'` and wipes `info`, `balance`, and `activity` to `null`/`null`/`[]`, even if the host never called `stop()` itself.
- **No automatic dispose**: the provider never calls `client.dispose()` on unmount, even for a client it built itself via `createClient`. The host is responsible for disposing the client when it is truly done with it.
</Callout>

<Callout type="note">
**Top-level `error` vs. `operations[op].error`.** The top-level `error` field is reserved for runtime-lifecycle failures the provider cannot attribute to a specific call: the initial `ready()` rejection, or exhausting the sync auto-poll's 5-failure bailout. Every operation you call directly (`start`, `stop`, `createWallet`, `unlockWallet`, `deposit`, `receive`, `send`, and the shared `refresh`) reports its own failure on `operations[op].error` instead, leaving the top-level `error` untouched.

**`info` is assembled incrementally.** Right after `createWallet`/`unlockWallet`, only `identityPubKey`, `walletState`, and `walletReady` are guaranteed to be set (the provider patches just those fields optimistically). The remaining `WalletInfo` fields (version, commit, network, block height, and so on) arrive once the follow-up `refresh()` completes, which is why `info` is typed `Partial<WalletInfo> | null` rather than `WalletInfo | null`.

**`refresh` is one shared operation.** Calling `useWalletDK().refresh()` and the `refresh` returned from `useWalletBalance()`/`useWalletActivity()` all invoke the exact same function and read the exact same `operations.refresh` busy/error slot, so a refresh triggered from one hook is visible as `busy` in every other hook that shares it.
</Callout>

</ApiSymbol>

## Re-exported from walletdk-core

`@lightninglabs/walletdk-react` re-exports the full `@lightninglabs/walletdk-core` surface (`export * from '@lightninglabs/walletdk-core'`), so `WalletDKClient`, `RuntimeConfig`, `defaultConfig`, `WalletInfo`, `WalletStatus`, `Balance`, `Entry` and the rest of the activity/exit/send type family, `RuntimePhase`, `WalletState`, and every request/result/enum type are all importable directly from this package. The enumerated values for `RuntimePhase` and `WalletState` are documented on the [walletdk-core reference](/reference/walletdk-core/) rather than repeated here.
