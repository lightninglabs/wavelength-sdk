# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

WalletDK is a TypeScript/React SDK for embedding a self-custodial Lightning wallet in a web app: send and receive Lightning payments with no node to run, no channels to open, and no inbound liquidity to manage. The full Go wallet daemon (sibling repo `../darepo-client`) is compiled to WebAssembly and runs in the browser; the SDK is a small, typed client over it. pnpm-workspace monorepo, pre-release.

## Commands

```sh
pnpm build        # build all packages (packages/*) in dependency order
pnpm typecheck    # typecheck every workspace project + scripts/
pnpm --filter @lightninglabs/walletdk-web test                                  # web unit tests (node --test)
pnpm --filter @lightninglabs/walletdk-web exec node --test src/passkey.test.ts  # a single test file
```

**Run `pnpm build` before `pnpm typecheck`.** The workspace typecheck resolves cross-package imports (e.g. `web` → `core`) through each package's built `dist/*.d.ts`; there are no tsconfig project references, so a fresh checkout has nothing to resolve until `build` runs. CI does build-then-typecheck for this reason.

### Demo + end-to-end smoke test

```sh
# 1. Build the wasm runtime into apps/web-wallet-demo/public/ (gitignored, ephemeral).
#    Needs Go + the sibling ../darepo-client checkout.
DAREPO_DIR="$(cd ../darepo-client && pwd)" pnpm --filter web-wallet-demo run wasm:local
pnpm --filter web-wallet-demo run build     # vite build; copies public/ -> dist/
pnpm --filter web-wallet-demo run test      # Playwright smoke test (headless Chromium)
```

The smoke test is **hermetic**: `apps/web-wallet-demo/smoke-server.js` mocks every backend and serves the built demo, so no regtest network is required. It defaults to port 8790 (the regtest stack occupies 7071/8501/8091/10032; `reuseExistingServer` will otherwise latch onto a stray service). It runs locally only. **CI does not run it.**

### Type generation

`packages/core/src/generated.ts` is auto-generated by `pnpm gen:types` (tygo) from the daemon's Go facade in `../darepo-client`. Do not hand-edit it; regenerate only when the daemon's facade types change (needs Go + tygo + the darepo-client checkout).

## Architecture

The design has **two orthogonal axes**: *transport* (how you reach the wallet daemon) × *framework binding* (how a UI framework integrates). Both rest on the `core` contract, and keeping them decoupled is the central constraint.

- **`packages/core`** (`@lightninglabs/walletdk-core`) is the transport- and framework-agnostic contract: the `WalletDKClient` interface, request/response types, `RuntimeConfig` + `defaultConfig`, errors, enums, and the generated daemon types. No DOM, no transport. Re-exported by every transport.
- **`packages/web`** (`@lightninglabs/walletdk-web`) is the browser **wasm transport** implementing `WalletDKClient`. The daemon runs in a Web Worker by default; `createWebClient()` is the factory. `src/clients/{base,main,worker}.ts` are the client classes, `runtime.ts` loads the wasm/asset bundle, `mobile-config.ts` maps `RuntimeConfig` onto the daemon config, and the WebAuthn passkey ceremony (`webPasskeyCeremony`) lives here. Re-exports `core`.
- **`packages/react`** (`@lightninglabs/walletdk-react`) is a **transport-agnostic** provider + hooks. It depends only on `core` and takes an **injected client** via `<WalletDKProvider client={…}>`. This injected-client seam is load-bearing: it keeps the React layer reusable across the web transport today and a future React-Native (gomobile) transport, and lets the web transport also drive a future Vue/Svelte binding. **Do not reintroduce a `walletdk-web` dependency in `react`.**
- **`apps/web-wallet-demo`** is the reference app plus the smoke test. Not published.

Cross-cutting:
- Daemon JSON is normalized at the transport boundary: PascalCase → camelCase (`camelizeKeys`), and the proto-numeric `WalletState` → a lowercase string union (`normalizeInfo` / `walletStateFromProto`).
- Passkey ceremonies are injected (`PasskeyCeremony` contract in `core`, `webPasskeyCeremony` in `web`); `usePasskeyWallet(ceremony)` takes it as an argument so `react` stays transport-free.
- The wasm runtime assets in `apps/web-wallet-demo/public/` are gitignored and rebuilt from `../darepo-client`. A versioned CDN default for `runtimeBaseUrl` is planned but not yet wired, so self-host for now.

## Conventions and preferences

**Git history.** Keep it a clean series of net-new logical commits, each building correctly on its own, not a "create then refactor" layer. When changing work that already exists on the branch, **fold the change into the commit that introduced it** (a `hunk rebase` fixup) rather than stacking a new commit; force-push is fine on the feature branch. Use `hunk` for non-interactive precision staging and fixup-rebases. Verify each commit builds.

**Commit messages.** Bare area prefix, no conventional-commit type wrapper: `core:`, `web:`, `react:`, `demo:` (and `chore:` / `docs:` for non-package work). Descriptive, imperative subjects.

**Public docs.** Consumer-facing docs (the README) pitch the product value: a self-custodial Lightning wallet with no node/channels/liquidity to manage. The underlying protocol (Ark) and the daemon repo are implementation details: keep them out of the README and other public copy. (Internal docs like this file may name them.)

**No em-dashes.** Do not use the em-dash character (U+2014) anywhere in the repo: comments, TSDoc, markdown, UI copy, config, all of it. When the prose wants that pause, reach for a colon, semicolon, comma, parentheses, or two sentences, whichever reads naturally. Generated prose tends to emit em-dashes, so self-check before committing (`git grep` for the character).

**TSDoc.** This is an open-source SDK, so document the public surface with `/** … */` TSDoc (the `WalletDKClient` interface, exported types and their fields, the provider/hooks, the factory and its options). Put the doc on the consumer-facing type, not on internal implementations.

**Code organization.** Split large files into per-concern modules, but don't over-organize: a flat `src/` of well-named files is fine until a package grows, and subdirectories are only worth it for genuine clusters (e.g. `web/src/clients/`). Drop redundant filename suffixes when the directory already supplies the context.

**Verification.** Treat the Playwright smoke test as the gold-standard check: it exercises the whole stack (create wallet → boarding address → Lightning invoice → activity → reload/unlock for OPFS persistence) in a real browser.
