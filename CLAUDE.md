# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

WalletDK is a TypeScript/React SDK for embedding a self-custodial Lightning wallet in a web app: send and receive Lightning payments with no node to run, no channels to open, and no inbound liquidity to manage. The full Go wallet daemon (sibling repo `../darepo-client`) is compiled to WebAssembly and runs in the browser; the SDK is a small, typed client over it. pnpm-workspace monorepo, pre-release.

## Commands

```sh
pnpm build        # build all packages (packages/*) in dependency order
pnpm typecheck    # typecheck every workspace project + scripts/
pnpm --filter @lightninglabs/walletdk-web test                                  # web unit tests (node --test)
pnpm --filter @lightninglabs/walletdk-web exec node --test src/passkey.test.ts  # a single test file
pnpm --filter @lightninglabs/walletdk-react-native test                         # RN transport unit tests (node --test)
pnpm --filter @lightninglabs/walletdk-react-native run fetch-bindings           # stage gomobile binaries; needs DAREPO_DIR
npx expo run:android   # from apps/rn-wallet-demo; builds and launches the dev client on an emulator/device
npx expo run:ios       # same, for the iOS simulator/device
```

**Run `pnpm build` before `pnpm typecheck`.** The workspace typecheck resolves cross-package imports (e.g. `web` â†’ `core`) through each package's built `dist/*.d.ts`; there are no tsconfig project references, so a fresh checkout has nothing to resolve until `build` runs. CI does build-then-typecheck for this reason.

### Demo + end-to-end smoke test

```sh
# 1. Build the wasm runtime into apps/web-wallet-demo/public/runtime/<version>/ (gitignored, ephemeral).
#    Needs Go + the sibling ../darepo-client checkout.
DAREPO_DIR="$(cd ../darepo-client && pwd)" pnpm --filter web-wallet-demo run wasm:local
pnpm --filter web-wallet-demo run build     # vite build; copies public/ -> dist/
pnpm --filter web-wallet-demo run test      # Playwright smoke test (headless Chromium)
```

The smoke test is **hermetic**: `apps/web-wallet-demo/smoke-server.js` mocks every backend and serves the built demo, so no regtest network is required. It defaults to port 8790 (the regtest stack occupies 7071/8501/8091/10032; `reuseExistingServer` will otherwise latch onto a stray service). It runs locally only. **CI does not run it.**

### React Native demo (manual verification)

`apps/rn-wallet-demo` has no hermetic smoke test; there is no wasm-style
mock target for a compiled-in native runtime. Verification is device-driven:
build and run the dev client with `npx expo run:android` / `npx expo
run:ios` from that directory and exercise the flows by hand against a
regtest stack. Before either command, stage the native binaries with
`pnpm --filter @lightninglabs/walletdk-react-native run fetch-bindings`
(needs `DAREPO_DIR`; see the package README).

Metro's dev-server cache can serve a stale JS bundle after native code or
dependency changes, a real recurring issue during bring-up. If the app on
device looks like it is running old code, restart Metro with `--clear`
rather than chasing the symptom.

### Type generation

`packages/core/src/generated.ts` is auto-generated by `pnpm gen:types` (tygo) from the daemon's Go facade in `../darepo-client`. Do not hand-edit it; regenerate only when the daemon's facade types change (needs Go + tygo + the darepo-client checkout).

## Architecture

The design has **two orthogonal axes**: *transport* (how you reach the wallet daemon) Ă— *framework binding* (how a UI framework integrates). Both rest on the `core` contract, and keeping them decoupled is the central constraint.

- **`packages/core`** (`@lightninglabs/walletdk-core`) is the transport- and framework-agnostic contract: the `WalletDKClient` interface, request/response types, `RuntimeConfig` + `defaultConfig`, errors, enums, and the generated daemon types. No DOM, no transport. Re-exported by every transport. It also carries the transport-shared implementation: `base-client.ts`'s `BaseWalletDKClient` implements every RPC verb once in terms of an abstract `callRaw`, so a transport supplies only its pipe (callRaw, ready, activity-stream plumbing, and a `ServerTransport` flavor), and `facade.ts` maps `RuntimeConfig` onto the daemon's wire config, keyed by `ServerTransport` (`'rest'` for the web transport, which cannot dial native gRPC; `'grpc'` for native transports).
- **`packages/web`** (`@lightninglabs/walletdk-web`) is the browser **wasm transport** implementing `WalletDKClient`. The daemon runs in a Web Worker by default; `createWebClient()` is the factory. `src/clients/{main,worker}.ts` are the client classes, `runtime.ts` loads the wasm/asset bundle, and the WebAuthn passkey ceremony (`webPasskeyCeremony`) lives here. Re-exports `core`.
- **`packages/react-native`** (`@lightninglabs/walletdk-react-native`) is the **React Native transport** implementing `WalletDKClient`: a hand-authored New-Architecture-only Turbo Module (not scaffolded with create-react-native-library) wrapping the gomobile-compiled wallet daemon. `createNativeClient()` is the factory; `src/client.ts`'s `NativeWalletDKClient` extends `BaseWalletDKClient` and is a thin JSON pipe over the native module's `call`, re-emitting native `walletdkActivity` device events as typed `activity` events. The native binaries (`android/libs/Walletdk.aar`, `ios/Walletdk.xcframework`) are gitignored and staged by `scripts/fetch-bindings.sh` from a darepo-client checkout, not committed. Re-exports `core`.
- **`packages/react`** (`@lightninglabs/walletdk-react`) is a **transport-agnostic** provider + hooks. It depends only on `core` and takes an **injected client** via `<WalletDKProvider client={â€¦}>` or via the `createClient` factory prop (client wins when both are given). This injected-client seam is load-bearing: it keeps the React layer reusable across the web and React Native transports, and lets the web transport also drive a future Vue/Svelte binding. **Do not reintroduce a `walletdk-web` dependency in `react`.**
- **`apps/web-wallet-demo`** is the reference web app plus the smoke test. Not published.
- **`apps/rn-wallet-demo`** is the reference React Native app: an Expo dev-client demo (New Architecture enabled). Its `android/` and `ios/` directories are generated by `expo prebuild` (implicitly, via `expo run:*`) and gitignored, same treatment as the wasm runtime assets. Not published.

Cross-cutting:
- Daemon JSON is normalized at the transport boundary: PascalCase â†’ camelCase (`camelizeKeys`), and the proto-numeric `WalletState` â†’ a lowercase string union (`normalizeInfo` / `walletStateFromProto`).
- Passkey ceremonies are injected (`PasskeyCeremony` contract in `core`, `webPasskeyCeremony` in `web`); `usePasskeyWallet(ceremony)` takes it as an argument so `react` stays transport-free.
- The wasm runtime assets in `apps/web-wallet-demo/public/runtime/<version>/` are gitignored and rebuilt from `../darepo-client`. The version segment is `RUNTIME_MANIFEST_VERSION` (`packages/core/src/version.ts`), the darepo-client revision the SDK is paired with; bump it together with regenerating types and publishing new hosted assets, and the versioned URLs bust browser caches. A versioned CDN default for `runtimeBaseUrl` is planned but not yet wired, so self-host for now.

## Conventions and preferences

**Git history.** Keep it a clean series of net-new logical commits, each building correctly on its own, not a "create then refactor" layer. When changing work that already exists on the branch, **fold the change into the commit that introduced it** (a `hunk rebase` fixup) rather than stacking a new commit; force-push is fine on the feature branch. Use `hunk` for non-interactive precision staging and fixup-rebases. Verify each commit builds.

**Commit messages.** Bare area prefix, no conventional-commit type wrapper: `core:`, `web:`, `react:`, `demo:` (and `chore:` / `docs:` for non-package work). Descriptive, imperative subjects.

**Public docs.** Consumer-facing docs (the README) pitch the product value: a self-custodial Lightning wallet with no node/channels/liquidity to manage. The underlying protocol (Ark) and the daemon repo are implementation details: keep them out of the README and other public copy. (Internal docs like this file may name them.)

**No em-dashes.** Do not use the em-dash character (U+2014) anywhere in the repo: comments, TSDoc, markdown, UI copy, config, all of it. When the prose wants that pause, reach for a colon, semicolon, comma, parentheses, or two sentences, whichever reads naturally. Generated prose tends to emit em-dashes, so self-check before committing (`git grep` for the character).

**TSDoc.** This is an open-source SDK, so document the public surface with `/** â€¦ */` TSDoc (the `WalletDKClient` interface, exported types and their fields, the provider/hooks, the factory and its options). Put the doc on the consumer-facing type, not on internal implementations.

**Code organization.** Split large files into per-concern modules, but don't over-organize: a flat `src/` of well-named files is fine until a package grows, and subdirectories are only worth it for genuine clusters (e.g. `web/src/clients/`). Drop redundant filename suffixes when the directory already supplies the context.

**Verification.** Treat the Playwright smoke test as the gold-standard check: it exercises the whole stack (create wallet â†’ boarding address â†’ Lightning invoice â†’ activity â†’ reload/unlock for OPFS persistence) in a real browser.
